INTRODUCTION AUX TECHNIQUES DE L’INFORMATION ET DE LA COMMUNICATION
1. Les techniques de l’information et de la communication (TIC) recouvrent tous les nouveaux moyens et outils qui permettent de traiter (matériels et logiciels de traitement), transmettre (réseaux et moyens de transmission et d’échange), de conserver (les supports de stockage) l’information électronique.
Les TICs recouvrent tous les nouveaux moyens et outils permettant la transmission de l’information à l’aide des réseaux et les moyens d’échange, le traitement de l’information, la sauvegarde de l’information dans les supports de stockage.
Les TICs offrent aux utilisateurs plusieurs services comme : la téléphonie vocale, transmission de données informatiques, communication par internet, diffusion multimédia et audiovisuel.
2. Catégories et objectifs
Les TIC sont présentés dans trois catégories : - La téléinformatique - Télécommunication - L’électronique
On peut résumer l’apport immédiat des TIC en la connectivité, la modularité, une implantation simple et une utilisation facile.
3. Composants et caractéristiques
Les TIC sont composées d’une part, de la partie matérielle (pc portable, téléphone portable les robots.. etc ) et de station de travail (PC, MAC, SUN, TERMINAL X), d’autre part de la partie logiciel (système d’exploitation, logiciels d’application) et des réseaux d’internet ou mobiles.
Les caractéristiques des TIC se résument en la rapidité (compression de temps), la décentralisation du pouvoir (flexibilité d’usage), accès performant aux informations, rationalité, la mondialisation vu l’activité des de la mondialisation économique à laquelle l’entreprise participe et en fin la répartition de l’information sur plusieurs échelles.

Système de gestion de Version
1/ Un système de gestion de version permet de suivre les modifications de fichiers et de projets logiciels. Ses utilisations principales sont :
1. Historique des modifications : Il enregistre les changements effectués et permet de revenir à des versions antérieures.
2. Collaboration en équipe : Il facilite le travail en équipe en fusionnant les modifications de manière ordonnée.
3. Récupération en cas de problème : Il permet de revenir à des versions antérieures en cas d'erreurs.
4. Gestion des branches : Il permet de travailler sur des fonctionnalités séparées du code principal.
5. Suivi des responsabilités : Il enregistre les contributions individuelles pour la revue du code.
6. Déploiement et livraison : Il facilite la livraison continue de logiciels.
En résumé, un système de gestion de version assure un suivi des modifications, facilite la collaboration, permet de revenir en arrière, gère les branches, assure le suivi des responsabilités et facilite le déploiement.
2/ Modèle centralisé vs modèle décentralisé :
Le modèle centralisé des logiciels de gestion de version repose sur un serveur central qui détient l'historique et les fichiers sources. Les utilisateurs récupèrent les fichiers du serveur, effectuent leurs modifications et les renvoient au serveur. Cela crée une structure hiérarchique où toutes les modifications sont centralisées.
En revanche, le modèle décentralisé des logiciels de gestion de version fonctionne sans un serveur central unique. Chaque utilisateur dispose d'une copie complète du référentiel, y compris l'historique. Les modifications sont propagées entre les référentiels, permettant un travail indépendant et la fusion ultérieure des modifications.
En résumé, le modèle centralisé repose sur un serveur central pour le stockage et le suivi des modifications, tandis que le modèle décentralisé permet aux utilisateurs d'avoir leurs propres référentiels et de fusionner les modifications entre eux.
3/ qu’est-ce que GitHub :
GitHub est une plateforme de développement collaboratif basée sur Git. C'est un service en ligne qui permet aux développeurs de stocker, gérer et partager leurs projets logiciels. Il facilite la collaboration en équipe, offre des fonctionnalités de contrôle d'accès, intègre des outils d'intégration continue et dispose d'une vaste communauté de développeurs. GitHub est devenu un élément essentiel de l'écosystème du développement logiciel, favorisant le partage des connaissances et la collaboration entre les développeurs du monde entier.
4/ L’utilisation de GitHub :
Git est un outil de contrôle de version utilisé dans le développement de logiciels. On peut l'utiliser de différentes manières : en ligne de commande, via une console ou une interface graphique. La ligne de commande offre le contrôle total sur Git, mais nécessite de saisir des commandes spécifiques. La console Git est plus conviviale et permet d'effectuer les actions courantes. L'interface graphique fournit une approche visuelle pour gérer les dépôts Git. Choisissez la méthode qui convient le mieux à vos besoins et à votre niveau de compétence en programmation

. Une fois Git installé, vous devez configurer certaines informations telles que votre nom d'utilisateur et votre adresse e-mail pour qu'elles soient enregistrées et utilisées par Git. Voici comment le faire : 6/ Paramétrage de Git :

8/ La gestion des informations selon Git :
Git gère les informations sous forme d'instantanés, en enregistrant l'état du projet à chaque validation. Les instantanés sont stockés localement, ce qui permet d'effectuer la plupart des opérations Git sans connexion à un serveur distant, rendant le travail plus rapide et agréable. 9/ Les états des fichiers :
.Un fichier peut avoir deux grands états dans Git: suivi ou non suivi.
.Un fichier suivi appartient au dernier instantané capturé par Git et est enregistré en base.
.Un fichier non suivi n'appartenait pas au dernier instantané et n'a pas été indexé. ors du démarrage d'un dépôt Git, les fichiers sont au départ tous non suivi et doivent être indexés et validés. .Chaque fichier suivi peut avoir l'un de ces trois états: modifié, indexé, validé.
.Un fichier est modifié par Git tant qu'il n'a pas été indexé.
. On indexe un fichier pour indiquer à Git que le fichier modifié ou le nouveau fichier doit faire partie du prochain instantané dans sa version actuelle.
.Lorsqu'on demande à Git de prendre l'instantané, les fichiers faisant partie de l'instantané sont considérés comme validés à nouveau et le cycle peut recommencer. 10 / Les zones de travail :
.Les fichiers dans Git sont liés à des zones de travail en fonction de leur état.
.Tout projet Git est composé de trois sections : le répertoire de travail, la zone d'index et le répertoire Git.
.Le répertoire de travail correspond à une extraction unique d'une version du projet.
.La zone d'index stocke les informations concernant ce qui fera partie du prochain instantané ou du prochain commit.
.Le répertoire Git est l'endroit où Git stocke les méta-données et la base de données des objets de votre projet.
.Le processus de travail consiste à travailler sur les fichiers dans le répertoire de travail, indexer les fichiers modifiés ou créés, valider la version indexée des fichiers pour les ajouter au répertoire Git. .Les états de fichiers dans Git sont modifié, non-suivi, et indexé.
.Des schémas sont disponibles sur git-scm.com pour mieux comprendre les zones de travail dans Git.
11/ créer un dépôt GiT à partir d'une répertoire existant ?
.Lorsque vous créez un dépôt Git à partir d'un répertoire existant en utilisant la commande "git init", Git crée un nouveau dossier caché .git dans le répertoire. Ce dossier contient toutes les informations nécessaires pour gérer les versions du projet, telles que : .L'historique des versions du projet.
.Les informations sur les commits, les branches et les tags.
.Les informations sur les remotes (dépôts distants).
.Les hooks (scripts personnalisés qui s'exécutent à des moments spécifiques).
.Les configurations Git pour le projet. Une fois que le dépôt Git est créé, vous pouvez utiliser les commandes Git pour ajouter, modifier et supprimer des fichiers, créer des branches et des tags, et enregistrer des versions dans le dépôt Git. Vous pouvez également ajouter des remotes pour travailler sur le projet avec d'autres développeurs et partager les modifications apportées au projet. .Et Pour créer un dépôt Git à partir d'un répertoire existant, il se fait de suivre ces étapes là :
1.Ouvrez votre terminal ou votre invite de commandes.
2. Naviguez jusqu'au répertoire que vous souhaitez transformer en dépôt Git.
3. Utilisez la commande "git init" pour initialiser un nouveau dépôt Git dans le répertoire.
4.Git va alors créer un nouveau dossier caché .git qui contiendra toutes les informations nécessaires pour gérer les versions du projet.
5. Utilisez la commande "git add" pour ajouter tous les fichiers du répertoire dans la zone d'index de Git.
6. Utilisez la commande "git commit" pour enregistrer tous les fichiers ajoutés dans la zone d'index dans le dépôt Git.
7. Le dépôt Git est maintenant créé et vous pouvez commencer à travailler sur le projet en utilisant les commandes Git pour gérer les versions.

Pour cloner un dépôt Git, il faut suivre ces étapes :
1.Ouvrez votre terminal ou votre invite de commandes.
2. Naviguez jusqu'au dossier où vous souhaitez cloner le dépôt Git.
3. Utilisez la commande "git clone" suivie de l'URL du dépôt Git que vous souhaitez cloner.

13 / Ajouter ou modifier des fichiers dans un projet et actualiser notre dépôt Git : Pour ajouter, modifier ou supprimer des fichiers dans un projet, on utilise les commandes git add et git commit.Ces commandes permettent de conserver un historique des versions du projet. Lorsqu'on utilise git add, le commit se basera sur l'état du fichier au moment du git add. Si on modifie à nouveau le fichier après le git add, il faut effectuer un nouveau git add pour enregistrer les modifications dans le dépôt Git. Pour mettre en un coup les fichiers modifiés et déjà sous suivi dans la zone d'index puis pour les valider, on peut utiliser git commit avec l'option -a. L'option -a permet de passer outre la commande git add.
14 / Consulter l'histoire des modifications Git :
La commande "git log" permet de consulter l'historique des modifications Git. Cette commande affiche la liste des commits réalisés du plus récent au plus ancien. Par défaut, chaque commit est affiché avec sa somme de contrôle SHA-1, le nom et l'e-mail de l'auteur, la date et le message du commit. "git log" est la manière la plus simple de consulter l'historique des modifications Git. 15 / Annuler les modifications apportées à un fichier :
Un des avantages de la gestion de version est de pouvoir revenir à un état antérieur d'un projet. Après un commit, on continue à travailler sur les fichiers et parfois on voudra revenir à l'état précédent. On peut utiliser la commande "git checkout -- nom-du-fichier" ou "git restore" pour annuler les modifications apportées à un fichier et revenir à l'état du dernier instantané enregistré. Ces commandes permettent de "roll back" les modifications et de retrouver l'état précédent d'un fichier dans Git. 16 / qu'est-ce qu'une branche :
Une branche est une copie d'un projet pour développer de nouvelles fonctionnalités sans impacter le projet de base. Git a une approche différente pour les branches, une branche est simplement un pointeur vers un commit. Un commit est un objet qui contient un pointeur vers l'instantané du contenu indexé et des pointeurs vers le ou les commits précédant directement le commit courant. La branche par défaut dans Git s'appelle master et se déplace automatiquement à chaque nouveau commit pour pointer sur le dernier commit effectué. Une branche n'est pas une branche spéciale pour Git et peut être renommée. Créer une nouvelle branche dans Git crée simplement un nouveau .pointeur plutôt que de recopier l'intégralité du répertoire de travail.
